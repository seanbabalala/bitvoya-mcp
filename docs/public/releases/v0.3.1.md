# Bitvoya MCP v0.3.1

Session runtime stability and observability patch for the hosted Bitvoya MCP endpoint.

This release keeps the `v0.3.0` remote transport model, but fixes the main operational blind spots we found while validating Hermes-style agent flows against the hosted endpoint.

Hosted endpoint:

- `https://bitvoya.com/api/mcp`

## Highlights

- active MCP requests are no longer eligible for idle-session sweep cleanup
- session runtime state now tracks request counts and active in-flight request count
- recent closed session summaries are retained briefly for postmortem lookup
- `Session not found` responses now include structured diagnostics when the server knows why the session disappeared
- new structured logs make it easier to distinguish:
  - stale client session reuse after restart
  - idle timeout cleanup
  - explicit delete teardown
  - request-time failures

## Why This Release Matters

During hosted-agent testing, a repeated failure mode showed up as generic client-side `Session terminated` or `Session not found` errors.

Two different problems can produce that same symptom:

- the server legitimately cleaned up an idle session
- the client kept using a session id that no longer exists after restart or teardown

`v0.3.1` makes those cases visible in server logs and prevents idle cleanup from racing with long-running in-flight tool calls.

## Runtime Changes

- session manager now records:
  - `requestCount`
  - `activeRequestCount`
  - `lastRequestStartedAt`
  - `lastRequestCompletedAt`
  - `closedAt`
- idle sweep skips runtimes whose `activeRequestCount > 0`
- recently closed session metadata is retained for short-lived lookup diagnostics

## Observability Improvements

- `mcp_session_runtime_closed` now logs close reason plus lifecycle timing fields
- `mcp_session_lookup_miss` is emitted when a client references a missing `mcp-session-id`
- `404 Session not found` JSON-RPC responses can now include structured `error.data` diagnostics
- request logs now include session-close context when available

## Validation

Validated locally with:

- `npm run check:syntax`
- `npm run test:transport`
- `bash ./scripts/publish_registry.sh --validate-only`

Transport tests now additionally cover:

- lookup-miss diagnostics for recently closed sessions
- protection against idle-sweep cleanup while a request is still in flight
