# Bitvoya MCP v0.3.0

Remote transport compatibility release for the hosted Bitvoya MCP endpoint.

This version focuses on making the public Streamable HTTP path more stable across Cherry Studio, Hermes, and other generic MCP clients, while preserving the existing hosted auth and secure checkout model.

Hosted endpoint:

- `https://bitvoya.com/api/mcp`

## Highlights

- session-scoped runtime reuse for remote Streamable HTTP transport
- lower hot-path initialization cost for repeated requests in the same MCP session
- broader compatibility for Cherry Studio, Hermes, and generic MCP clients
- structured request diagnostics without leaking bearer tokens
- idle session cleanup plus explicit `DELETE` teardown
- new transport tests covering both Cherry-like and Hermes-like request flows

## Why This Release Matters

Previous remote behavior rebuilt the MCP runtime on every HTTP request. That was tolerable for some clients, but it increased first-hop latency and made session-oriented clients less stable.

`v0.3.0` moves the hosted transport closer to how real MCP clients behave in practice:

- initialize once
- reuse the same session runtime
- tolerate more real-world `Accept` header variations
- preserve Cherry-style SSE compatibility
- respond more predictably to generic JSON-oriented MCP clients

## Compatibility Improvements

- same `mcp-session-id` now reuses one connected runtime instead of rebuilding on every request
- `POST` responses use direct JSON response mode for better generic MCP client compatibility
- `GET` SSE flow remains intact for Cherry-like clients
- `OPTIONS`, `GET`, `POST`, and `DELETE` behavior remains available on the hosted endpoint

## Security And Observability

- remote auth behavior is preserved
- bearer tokens and raw `Authorization` headers are not written to logs
- request logs now capture method, path, user-agent, accept, content-type, origin, host, session, protocol version, JSON-RPC method, auth summary, latency, and runtime reuse state

## Validation

Validated locally with:

- `npm run check:syntax`
- `npm run test:transport`
- `npm run smoke:transport`

Transport tests cover:

- `OPTIONS` handling
- missing bearer auth behavior
- session runtime creation and reuse
- explicit `DELETE` cleanup
- idle timeout cleanup
- `Accept` header compatibility
- token-safe logging
- Cherry-like flow
- Hermes-like `initialize -> tools/list` flow

## Recommended Usage

- endpoint: `https://bitvoya.com/api/mcp`
- auth: `Authorization: Bearer <your_agent_key>`
- preferred client models: higher-capability models with reliable tool use, such as `Claude 4.6` and `GPT-5.4`
